# ARIMA: интеллектуальная система управления учебными задачами с AI-напоминаниями и родительским контролем

## Краткое резюме

Arima — это полноценное fullstack-решение для студентов, учителей и родителей, которое автоматизирует отслеживание домашних заданий, проектов и дедлайнов. Система использует AI для анализа продуктивности, интеллектуальных напоминаний и автоматического эскалирования просроченных задач родителям. Проект реализован с нуля: backend на FastAPI с асинхронной архитектурой, frontend на Flutter с кроссплатформенной поддержкой (Web + Mobile).

---

## Проблема, которую мы решаем

Каждый студент сталкивается с одной и той же болью: десятки задач, разбросанных по разным платформам, дедлайны, которые внезапно оказываются «вчера», и полное отсутствие обратной связи от учителей в реальном времени. Родители, в свою очередь, узнают о проблемах слишком поздно — когда уже ничего нельзя исправить.

Существующие решения (Google Classroom, Trello, Notion) либо слишком универсальны и не адаптированы под образовательный процесс, либо не имеют механизма вовлечения родителей. Мы решили создать узкоспециализированный инструмент, который закрывает именно эти пробелы.

## Архитектура решения

### Стек технологий

**Backend:**
- **FastAPI** — современный асинхронный фреймворк для REST API на Python 3.12
- **SQLAlchemy 2.0** (async) — ORM с полной поддержкой асинхронных операций
- **aiosqlite** — асинхронный драйвер SQLite для development-режима
- **Pydantic v2** — валидация данных и сериализация
- **APScheduler** — фоновый планировщик для системы напоминаний
- **PyJWT + PBKDF2** — JWT-аутентификация с безопасным хешированием паролей

**Frontend:**
- **Flutter 3.22+** — кроссплатформенный UI (Web, iOS, Android)
- **Riverpod** — реактивное управление состоянием с автоматической инвалидацией кэша
- **GoRouter** — декларативная навигация с role-based редиректами
- **Dio** — HTTP-клиент с перехватчиками для JWT-токенов
- **SharedPreferences** — локальное хранилище сессии

**База данных:**
- SQLite (development) / PostgreSQL (production)
- Схема: пользователи, задачи, напоминания, метрики, связи учитель-студент, родитель-ребенок

## Ключевые фичи и реализация

### 1. Ролевая модель с изолированными интерфейсами

Система поддерживает три роли — Student, Teacher, Parent — каждая со своим интерфейсом и набором функций. Роутинг на фронтенде полностью автоматизирован: после логина пользователь автоматически попадает на свой дашборд, а попытка зайти на чужой — редиректит обратно.

Реализовано через GoRouter с `refreshListenable`, который слушает изменения `AuthController` (Riverpod AsyncNotifier). Когда сессия меняется — маршрут пересчитывается автоматически, без ручного управления навигацией.

### 2. Система интеллектуальных напоминаний (Core Feature)

Это сердце проекта. Каждая задача может иметь настраиваемое напоминание с параметрами:
- `remind_after_hours` — интервал между напоминаниями
- `max_missed_count` — лимит пропусков перед эскалацией
- `escalated_to_parent` — флаг эскалации

**Логика работы:**
1. Фоновый job (APScheduler) запускается каждые 15 минут
2. Для каждой невыполненной задачи проверяется: прошло ли `remind_after_hours` с последнего напоминания
3. Если да — инкрементируем `missed_count`, обновляем `last_reminded_at`
4. Если `missed_count >= max_missed_count` — устанавливаем `escalated_to_parent = true` и генерируем `parent_alert_message`
5. На фронтенде родитель видит красный баннер с описанием проблемы

**Техническая сложность:** SQLite хранит naive datetimes, а Python использует timezone-aware UTC. Пришлось реализовать нормализацию `_normalize(dt)` во всех сервисах, чтобы избежать `TypeError: can't compare offset-naive and offset-aware datetimes`.

### 3. Teacher-Student Linking и Assignments

Учитель может привязать студента по email и назначать ему задачи с обязательной/необязательной сдачей. Схема связи Many-to-Many через промежуточную таблицу `teacher_student_links`.

При назначении задачи через `POST /teacher/tasks` создается Task с `assigned_by_teacher_id` и `requires_submission` флагом. Студент видит эту задачу в своем списке с пометкой от учителя.

### 4. Система сдачи и оценивания (1-5 балльная шкала)

Студент может отправить текстовый ответ на задачу. Учитель получает это в своей панели «Submissions» и выставляет оценку по шкале 1-5 (школьный стандарт) с текстовым фидбеком.

**Важное решение:** изначально делали шкалу 0-100, но потом поняли, что в российской/европейской школе принята шкала 1-5. Переделали валидацию в Pydantic (`Field(ge=1, le=5)`) и заменили TextField на DropdownButtonFormField с фиксированными значениями.

На фронтенде студент видит статус сдачи: оранжевая плашка «Submitted» или зеленая «Graded» с конкретной оценкой и комментарием учителя.

### 5. Аналитика и метрики

Система собирает метрики в реальном времени:
- Completion Rate (процент выполненных задач)
- Streak (дни подряд с выполненными задачами)
- Heatmap активности по дням недели
- Overdue alerts

Все вычисления агрегируются на бэкенде через `metrics_service.py` с учетом timezone-проблем. На фронтене отображаются через `MetricCard` — адаптивные карточки с иконками, цветовой кодировкой и прогресс-барами.

### 6. Адаптивный UI с Material Design 3

Интерфейс перестроен на современных принципах:
- **DashboardHeader** — приветствие с emoji, текущая дата, quick stats в виде сетки
- **ModernTaskCard** — карточка задачи с цветовой кодировкой предмета (Math=blue, Science=green, History=orange, English=purple, Art=pink), сложностью (Easy/Medium/Hard), временем и очками
- **MetricCard** — универсальная метрика с иконкой, значением, подзаголовком и опциональным прогресс-баром
- **NavigationBar** — нижняя навигация с тремя вкладками (Tasks / Profile / Stats)

Все компоненты responsive — корректно отображаются и на мобильном, и на вебе.

## Технические вызовы и как мы их решали

### Вызов 1: MissingGreenlet в async SQLAlchemy
**Проблема:** После добавления поля `submission` в `TaskResponse`, Pydantic при сериализации триггерил lazy load `task.submission`. В async режиме SQLAlchemy это приводит к `MissingGreenlet: greenlet_spawn has not been called`.

**Решение:** Добавили `selectinload(Task.submission)` во все запросы, которые возвращают TaskResponse. Теперь связь загружается eagerly, без lazy loading.

### Вызов 2: Offset-naive vs offset-aware datetime
**Проблема:** SQLite не хранит timezone info, а все системные timestamp'ы генерируются в UTC. При сравнении `due_at` (naive) с `datetime.now(timezone.utc)` (aware) — TypeError.

**Решение:** Нормализующий хелпер `_normalize(dt)`, который делает `.replace(tzinfo=timezone.utc)` для naive дат перед любыми сравнениями.

### Вызов 3: Циклические импорты в Pydantic v2
**Проблема:** `TaskResponse` ссылается на `TaskSubmissionResponse`, а тот — обратно. Pydantic v2 требует `model_rebuild()` для forward references.

**Решение:** Forward reference через строку `"TaskSubmissionResponse | None" = None` в `TaskResponse` + `TaskResponse.model_rebuild()` после импорта.

## Заключение

Arima — это не просто таск-менеджер, а полноценная образовательная экосистема. Мы сфокусировались на реальных болях пользователей: студент получает структурированный инструмент для планирования, учитель — возможность назначать и проверять работы, родитель — прозрачность в успеваемости ребенка. Технически проект построен на современном стеке с акцентом на производительность (async everywhere), безопасность (PBKDF2 + JWT) и масштабируемость (SQLAlchemy позволяет легко перейти на PostgreSQL).
